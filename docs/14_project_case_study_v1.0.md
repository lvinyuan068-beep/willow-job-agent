## 1. 项目概述

Willow Job Agent 是一套面向个人求职场景的本地AI岗位决策系统。

项目从真实求职痛点出发：招聘平台岗位数量大、销售或虚假运营岗位识别成本高，不同JD需要重复修改简历和准备话术，且大模型容易虚构经历或使用未经核验的数据。

系统输入岗位名称、公司、薪资、城市、JD和补充信息后，优先输出：

1. 是否投递；

2. 是否需要修改简历；

3. 可复制的HR沟通话术；

4. 岗位定向面试问题与示例答案；

5. 短期可补齐技能及执行方案。

系统同时提供风险岗位过滤、公司公开信息检索、证据边界保护、定向简历正文、执行指南、结果留档和网页操作界面。

## 2. 项目背景与真实问题

求职过程中需要同时处理多个高频任务：

- 判断岗位是否为真实运营岗位；

- 识别销售、陌生开发、续费转化、无薪试岗和收费培训等风险；

- 判断学历、经验、薪资和城市是否满足个人约束；

- 决定使用哪一版简历；

- 根据JD修改经历顺序和关键词；

- 回复HR或招聘顾问；

- 准备岗位面试问题；

- 判断能力缺口是否能够短期补齐；

- 核验公司、劳动合同主体和岗位真实性；

- 避免AI把不准确的数据写进简历。

此前依靠人工在多个AI和招聘平台之间反复复制信息，判断口径不统一，也难以复盘哪些修改真正提高了简历反馈率。

因此，本项目的核心不是生成更多内容，而是建立一套证据安全、规则稳定、可以持续评测和迭代的求职决策流程。

## 3. 目标用户

当前核心用户为本人，具有以下特征：

- 2026届影视摄影与制作本科毕业生；

- 求职方向包括AI产品运营、AI数据或多模态内容测评、AIGC内容运营、AI应用运营及普通运营；

- 有用户活动、问卷回收、工单审核、客诉沟通和运营汇报实践；

- 能使用Excel、PPT、Premiere、After Effects、Photoshop及部分AIGC工具；

- 没有正式AI岗位任职经历；

- 没有独立编程能力，需要AI辅助完成技术实现；

- 需要正式劳动合同和五险一金；

- 排除销售、陌生开发、无薪试岗和收费岗位；

- 希望提高HR查看简历、发出面试邀请及最终录用的概率。

## 4. 产品目标与成功指标

### 4.1 产品目标

将岗位分析从一次性AI问答升级为结构化决策系统，实现：

- 同一套岗位判断口径；

- 明确的硬性淘汰条件；

- 有证据来源的简历表述；

- 针对岗位的面试答案；

- 可执行而非只有标题的补齐任务；

- 可重复运行和回归测试；

- 每次分析结果自动保存。

### 4.2 第一阶段业务成功指标

第一阶段最终成功标准不是“程序能够启动”，而是：

- 实际求职中获得多个与目标方向较匹配的面试机会；

- 或成功获得一个符合底线要求的Offer。

### 4.3 过程指标

后续持续记录：

- 投递数量；

- HR查看或回复数量；

- 有效沟通数量；

- 简历初筛通过数量；

- 面试邀请数量；

- 面试通过数量；

- Offer数量；

- 不同简历版本和岗位方向的反馈率。

当前尚未取得足够市场结果，因此不得宣称系统已经提高面试率或录用率。

## 5. 用户研究与需求定义

项目通过多轮个人访谈和求职资料审计，整理了以下信息：

- 城市与最低收入约束；

- 岗位方向优先级；

- 可接受和不可接受的工作条件；

- 个人能力矩阵；

- 实习案例；

- 项目与作品；

- 已确认的数据；

- 不允许使用的简历表述；

- 真实投递反馈；

- 招聘平台上的高频判断任务。

需求优先级最终确定为：

### P0：岗位决策

- 判断投不投；

- 识别销售和风险岗位；

- 检查经验、薪资、城市和劳动关系；

- 直接淘汰明显不可能岗位。

### P0：求职执行

- 生成岗位定向简历正文；

- 生成HR沟通话术；

- 生成岗位定向面试答案；

- 给出下一步行动。

### P1：证据安全

- 只使用证据库允许的经历；

- 区分个人贡献和团队结果；

- 阻止虚构数据；

- 明确个人项目、课程作业和任职成果的区别。

### P1：联网研究

- 通过Tavily搜索公司公开资料；

- 返回来源名称和链接；

- 支持人工补充小公司信息；

- 明确搜索结果不等同于官方工商核验。

### P2：持续优化

- 建立真实岗位评测集；

- 记录投递漏斗；

- 根据真实反馈调整评分规则和简历策略。

## 6. 核心产品方案

### 6.1 岗位输入

支持网页表单、TXT和JSON三种输入形式。

主要字段包括：

- 岗位名称；

- 公司名称；

- 招聘平台；

- 城市；

- 薪资；

- 工作地点；

- 岗位JD；

- HR沟通信息；

- 公司补充信息；

- 用户问题。

### 6.2 岗位决策

系统输出四种决策：

- APPLY：可以直接投递；

- APPLY_AFTER_EDIT：修改简历后投递；

- VERIFY_FIRST：先核实关键信息；

- REJECT：直接淘汰。

同时输出岗位档位、岗位判断分、真实岗位类型、硬过滤状态、风险标记和缺失信息。

岗位判断分用于候选岗位之间的排序，不代表真实统计概率。

### 6.3 硬过滤

已经实现的主要红线包括：

- 销售或陌生开发；

- 续费、成交或转化指标与绩效强绑定；

- 无薪试岗；

- 要求求职者付费；

- 明显超过候选人的经验硬门槛；

- 城市薪资低于生活底线；

- 劳动关系或外包主体存在关键疑问。

对于信息不足但不应直接淘汰的岗位，系统返回VERIFY_FIRST，而不是把未知条件当作已经满足。

### 6.4 证据安全层

证据库当前包含16个可复用内容块和10项禁止表述。

重点保护规则包括：

- 可以使用“审核2000+条工单”；

- 可以使用“活动触达800+”；

- 可以使用“个人新增65名活动用户微信”；

- 可以使用“现场指导完成80份问卷”；

- 不使用“重复咨询率从65%精准下降到32%”；

- 不使用未经正式审计的93%或100%满意度；

- 不把3套个人流程写成公司正式落地SOP；

- 不把设施安全客诉写成个人独立定责、赔付和整改闭环；

- 不把课程作业包装成正式商业项目；

- 不把AI辅助开发描述为个人独立编程。

### 6.5 定向简历

系统根据岗位类型选择基础方向，并生成可复制的定向简历正文。

当前已支持：

- CRM及用户运营；

- AIGC内容生产与质量运营；

- AI产品运营；

- AI数据或多模态测评；

- AI应用运营；

-普通运营。

简历包装遵循：

- 先展示最相关能力；

- 使用岗位关键词但不照抄JD；

- 使用“任务—行动—结果—边界”表达；

- 保留真实数量；

- 删除不可核验结果；

- 明确个人贡献；

- 不虚构工具熟练度。

系统还具备Word简历生成模块，但网页当前优先提供可复制文本，方便人工完成最终排版和复核。

### 6.6 面试准备

系统根据岗位生成至少8道高概率问题，并为每道问题输出：

- 示例答案；

- 使用证据；

- 可能追问；

- 注意事项；

- 问题意图；

- 质量状态。

当前已针对CRM和AIGC内容岗位增加专用答案，包括：

- 用户生命周期；

- 复购分析；

- LTV；

- RFM；

- 会员体系；

- 邮件营销；

- AI效率；

- AIGC作品介绍；

- 分镜与素材拆解；

- 内容质量定位；

- 提示词和版本管理；

- 内容数据复盘；

- 快速上手方案。

题库具备语义去重和答非所问检查，避免多个问题生成相同答案或残留其他岗位内容。

### 6.7 短期补齐与执行指南

系统不只输出“需要学习某项技能”，还生成：

- 优先级；

- 预计耗时；

- 推荐交付格式；

- 执行步骤；

- 可复制模板；

- 验收标准。

已覆盖定向简历、岗位作品索引、运营数据复盘、多模态内容评测、会员体系设计、邮件营销基础和电商CRM指标等任务。

### 6.8 联网研究

系统使用Tavily搜索公开网页资料，并把来源名称和链接写入运行说明。

联网能力用于：

- 搜索公司公开资料；

- 查找公司官网和招聘信息；

-辅助判断招聘主体；

- 为“为什么选择公司”等面试问题提供资料。

当前联网检索不能替代国家企业信用信息公示系统或企业官方文件，不得把搜索结果表述为已完成官方工商核验。

### 6.9 模型主备

系统采用：

- Kimi作为主模型；

- DeepSeek作为备用模型；

- 本地规则引擎作为可配置的降级方案。

当Kimi失败时自动尝试DeepSeek；两个在线模型均失败时，系统返回明确错误或按照配置切换到本地规则模式。

### 6.10 本地网页

本地网页地址为：

`http://127.0.0.1:4310`

网页支持：

- 填写岗位信息；

- 查看岗位决策；

- 查看简历修改结果；

- 复制HR话术；

- 查看逐题面试答案；

- 查看短期补齐任务；

- 查看证据和风险；

- 一键复制主要文本；

- 下载完整JSON；

- 自动保存latest和history结果。

## 7. 系统结构

系统主要分为五层：

1. 输入层：网页、TXT、JSON；

2. 决策层：硬过滤、岗位分类、排序评分；

3. AI与研究层：Kimi、DeepSeek、Tavily；

4. 执行层：定向简历、沟通话术、面试答案和补齐指南；

5. 质量层：证据保护、结构化契约、回归测试和结果留档。

当前工程快照包括：

- 17个TypeScript核心模块；

- 4个网页文件；

- 4个评测脚本；

- 本地网页、命令行和结构化JSON输出；

- Mock、Failover及联网运行模式。

代码主要由AI辅助生成和修复。本人负责需求定义、产品结构、规则设计、证据核对、功能验收、问题反馈、测试执行和迭代决策，不宣称独立编程开发能力。

## 8. 关键迭代过程

### 8.1 从长提示词转为模块化系统

最初需求集中在一段长提示词中，混合了目标、工作流、工具、异常处理和输出格式。

后续拆分为：

- Mission；

- User Profile；

- Constraints；

- Workflow；

- Input Contract；

- Output Contract；

- Evaluation；

- Failure Handling。

这一调整使需求从作文式描述转为可维护的产品和系统结构。

### 8.2 从通用AI回答转为证据约束

早期简历内容中出现过AI概括的精确比例和过度归因。

后续建立证据账本、能力矩阵和禁止表述清单，并在生成、结构化解析和真实岗位评测中重复检查。

### 8.3 从岗位建议转为求职执行包

早期系统只回答投不投、简历怎么改和面试问题标题，实际执行价值不足。

后续增加：

- 完整定向简历正文；

- 逐题示例答案；

- 证据与追问；

- 专项执行步骤；

- 交付模板；

- 验收标准；

- 一键复制。

### 8.4 从泛化答案转为岗位专用答案

真实CRM岗位测试中，部分答案过于通用。

随后增加CRM、会员、复购、LTV、RFM和邮件营销专项逻辑。

真实AIGC岗位测试中，初始面试答案通过率仅为37.5%，并出现CRM内容残留。

增加AIGC专用答案后，8道问题全部通过质量检查，真实岗位评测由1/2提升为2/2。

该结果是评测集内部通过情况，不等同于真实招聘录用率提升。

### 8.5 从功能可运行转为可回归验证

项目逐步建立四层测试：

- 本地规则评测；

- 模型故障切换评测；

- 网页端冒烟测试；

- 真实岗位端到端评测。

每次修改后重新运行测试，避免修复新功能时破坏已经通过的功能。

7. 将第一阶段成功标准绑定到有效面试机会或Offer，而不是单纯绑定程序测试结果。

## 15. 项目结论

Willow Job Agent 已从提示词概念发展为一个本地可运行、具备证据保护、在线模型、联网研究、网页交互和回归评测的求职AI产品原型。

当前可以用于辅助真实岗位筛选、简历定向修改、HR沟通和面试准备，但仍处于真实市场验证阶段。

项目当前最准确的状态描述是：

“完成了一个AI辅助搭建的本地求职决策Agent，并通过规则、故障切换、网页和两个真实岗位的端到端评测；正在通过真实投递反馈验证其对面试机会和Offer的实际提升效果。”
